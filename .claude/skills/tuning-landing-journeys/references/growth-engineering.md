## Growth Loops for Developer Libraries

ModulusKit's growth is compounding when adoption creates more adoption. Two loops apply:

**Loop 1: Scaffold → Share**
```
Developer runs `modulus init` →
  Generated solution contains ModulusKit references and README attribution →
    Teammate opens the repo, sees ModulusKit →
      Teammate evaluates and installs for their next project
```

Reinforce this loop by ensuring generated solutions include a visible attribution comment and link:

```csharp
// Generated: src/MyApp.AppHost/Program.cs (Aspire template)
// Scaffolded with ModulusKit CLI — https://github.com/AdamWyatt34/Modulus

var builder = DistributedApplication.CreateBuilder(args);
```

**Loop 2: Docs → Evangelize**
```
Developer reads recipe or architecture doc →
  Shares link in Slack/Discord/blog →
    New developer reads, evaluates, installs
```

Recipes (`docs/recipes/`) are the highest-leverage content for this loop because they solve specific, searchable problems ("modular monolith outbox pattern .NET").

## Acquisition Levers

### 1. NuGet Search Rank

The single highest-leverage acquisition channel is appearing in NuGet search for `modular monolith`, `cqrs dotnet`, and `mediator nuget`. This is pure SEO: `<PackageTags>` and `<Description>` content.

Current tag gaps to fill:

```xml
<!-- Directory.Build.props or individual .csproj files -->
<!-- Add these missing high-volume search terms: -->
<PackageTags>...; clean-architecture; bounded-context; ddd; domain-driven-design</PackageTags>
```

### 2. Docs Site SEO (docs/ VitePress)

Recipes and feature pages rank for long-tail search. Each recipe in `docs/recipes/` should target a specific query:

```markdown
<!-- docs/recipes/outbox-pattern.md -->
---
title: "Implementing Transactional Outbox Pattern in .NET Modular Monolith"
description: "Step-by-step guide to reliable event publishing with ModulusKit.Messaging outbox pattern."
---
```

The `title` and `description` frontmatter is rendered as `<title>` and `<meta name="description">` by VitePress — directly impacting search ranking.

### 3. GitHub Ecosystem Discoverability

GitHub Topics on the repository surface ModulusKit in topic browse pages:

```
Recommended topics to add via GitHub Settings → Repository → Topics:
- modular-monolith
- cqrs
- dotnet-tool
- dotnet
- mediator
- outbox-pattern
- aspire

```

### 4. CLI Attribution in Generated Code

Every file generated by `modulus init` or `modulus add-*` is an organic impression. Generated file headers:

```csharp
// src/Modulus.Templates/ — Scriban templates
// Add to top of generated handler files:
// Generated by ModulusKit CLI (https://github.com/AdamWyatt34/Modulus)
```

This surfaces to: code reviewers, GitHub repository searches, and developers who clone the generated solution.

## Retention Signals

Retention for a library = continued use across versions. Signals of churn:

- Issues mentioning migration difficulty between versions
- Forks with local patches (developer gave up on upstream updates)
- "Alternatives to ModulusKit" questions in issues

Retention levers in the codebase:

**CHANGELOG.md** — Developers decide to upgrade based on changelog quality. Current format is good; ensure each entry links to the relevant docs page:

```markdown
## [1.2.0] - 2026-03-07
### Added
- `add-endpoint` CLI command — [docs](/cli/add-endpoint)
- StronglyTypedId with EF Core converter generation — [docs](/generators/strongly-typed-ids)
```

**Upgrade path guidance** — Each major version needs a migration section in docs. Absence of this is a churn driver.

## Referral Mechanics

Developers refer libraries through:
1. Blog posts ("how I structure modular monoliths with ModulusKit")
2. Conference talks with code slides (generated solutions visible)
3. GitHub repos using ModulusKit (public repos that `dotnet add package ModulusKit.*`)
4. Community answers (Stack Overflow, Discord, Reddit /r/dotnet)

Enabling blog referrals:
- Ensure `docs/recipes/` pages are deep-linkable and have clear titles
- Add "Share this recipe" links (or just make URLs clean: `/recipes/outbox-pattern`)

Enabling community answers:
- Add `docs/faq/` with answers to common SO questions about modular monoliths
- Each FAQ entry is a potential SO answer with a canonical link

## Anti-Patterns

### WARNING: Prioritizing GitHub Stars Over NuGet Downloads

Stars are a vanity metric for libraries. A developer who stars a repo but never installs it contributes nothing to the growth loop. Focus copy and distribution work on NuGet install count (actual usage).

### WARNING: Growth Tactics That Skip the Quality Bar

**The Problem:**
Submitting ModulusKit to "awesome-dotnet" lists or Reddit before the getting-started experience is smooth creates net-negative impressions — the developer installs, hits friction, and publicly says "this isn't ready."

**The Fix:**
Before any proactive distribution push, verify the install-to-scaffold journey takes less than 5 minutes with zero prior knowledge. Run through it fresh in a clean environment.
