## Growth Levers for a Developer Library

ModulusKit's growth comes from two sources: **active discovery** (developers searching for a solution) and **passive distribution** (scaffolded projects pulling packages as dependencies). Both are engineering problems, not marketing problems.

| Growth Lever | Mechanism | Code Surface |
|-------------|-----------|-------------|
| NuGet search ranking | Package tags, download velocity | `.csproj` `<PackageTags>` |
| GitHub referral | Star count, README quality | `README.md` |
| Passive distribution | Scaffolded projects depend on `ModulusKit.*` | Template files in `src/Modulus.Templates/` |
| Developer advocacy | Clear attribution in generated code | `AddModulusHandlers()` generated extension |
| Analyzer noise | MOD001-005 daily touchpoints | `src/Modulus.Analyzers/` |

## CLI as a Growth Surface

Every `modulus init` execution is a potential growth event. The scaffolded solution contains NuGet references to all `ModulusKit.*` packages — those references propagate when the developer shares or open-sources their project.

The generated `Directory.Packages.props` is a passive distribution artifact:

```xml
<!-- Generated by: modulus init EShop -->
<!-- This file in a public repo exposes ModulusKit to anyone who forks or studies it -->
<Project>
  <ItemGroup>
    <PackageVersion Include="ModulusKit.Mediator" Version="1.x.x" />
    <PackageVersion Include="ModulusKit.Messaging" Version="1.x.x" />
    <PackageVersion Include="ModulusKit.Generators" Version="1.x.x" />
    <PackageVersion Include="ModulusKit.Analyzers" Version="1.x.x" />
  </ItemGroup>
</Project>
```

**Implication:** Every template file in `src/Modulus.Templates/` that references a `ModulusKit.*` package is a growth artifact. Keep package names visible and consistent.

## Adoption Loop via Scaffolded Code

The source generator in `src/Modulus.Generators/` produces `AddModulusHandlers()`. This method name is visible to every developer reading the generated project:

```csharp
// Generated: src/EShop.WebApi/Program.cs
// Every reader of this file sees the ModulusKit ecosystem
builder.Services.AddModulusMediator();
builder.Services.AddModulusHandlers(); // ← source-generated, visible attribution
builder.Services.AddModulusMessaging(options => { ... });
```

This is attribution without friction. NEVER rename these methods to generic names — `AddHandlers()` or `AddMediator()` loses the discovery signal.

## Package Dependency Graph as Distribution

Each package in the ModulusKit ecosystem should have a minimal, focused dependency surface. Developers evaluate transitive dependencies before adopting — a package that pulls in 15 transitive dependencies loses against one that pulls in 3.

```xml
<!-- src/Modulus.Mediator.Abstractions/Modulus.Mediator.Abstractions.csproj -->
<!-- Abstractions package MUST have zero transitive dependencies -->
<ItemGroup>
  <!-- No PackageReference items — pure interfaces and types only -->
</ItemGroup>
```

```xml
<!-- src/Modulus.Mediator/Modulus.Mediator.csproj -->
<!-- Implementation package has minimal deps: FluentValidation, MS.Extensions.DI -->
<ItemGroup>
  <PackageReference Include="FluentValidation" />
  <PackageReference Include="Microsoft.Extensions.DependencyInjection.Abstractions" />
</ItemGroup>
```

The `Abstractions` packages being dependency-free is a growth feature. Teams that can't use MassTransit can still adopt `ModulusKit.Mediator.Abstractions` for the Result pattern alone.

## Attribution in Error Messages

Roslyn analyzer diagnostics (MOD001-MOD005) show up in IDE error lists daily for adopting teams. Each diagnostic is a touchpoint. The rule ID `MOD` prefix is attribution:

```csharp
// src/Modulus.Analyzers/
// Rule ID format: MOD + three digits — "MOD" is visible in every IDE diagnostic
public static readonly DiagnosticDescriptor Rule = new(
    id: "MOD002",           // ← visible in IDE, searchable on GitHub
    title: "Handler must return Result",
    // ...
);
```

Developers who encounter `MOD002` and search for it will land on the Modulus GitHub repo. This is a passive acquisition channel — maintain it by keeping rule documentation up to date.

## Anti-Patterns

### WARNING: Generic Method Names in Generated Code

**The Problem:**

```csharp
// BAD — generated code with no attribution
builder.Services.AddHandlers();
builder.Services.AddMessaging();
```

**Why This Breaks:** When another developer reads this project, they have no way to know which library produced these extensions or how to find it. The growth loop is broken.

**The Fix:** Always use the `ModulusKit`-namespaced extension names in generated templates:

```csharp
// GOOD — visible attribution in every project
builder.Services.AddModulusMediator();
builder.Services.AddModulusHandlers();
```

### WARNING: Heavy Transitive Dependencies in Abstractions Packages

`ModulusKit.Mediator.Abstractions` and `ModulusKit.Messaging.Abstractions` must remain free of runtime dependencies. Adding a dependency here forces it on every downstream consumer, including those who only want the interfaces.

A team evaluating "should I adopt ModulusKit" will reject it if the abstractions package pulls in MassTransit or EF Core transitively. Abstractions packages are adoption gates.
