## Result Pattern

`Result` and `Result<TValue>` are the **only** return types for handlers. Never throw for expected failures.

```csharp
// src/Modulus.Mediator.Abstractions/Results/Result.cs

// Void success
return Result.Success();

// Value success (implicit conversion — preferred)
User user = await repo.FindAsync(id, ct);
return user;                                     // implicit operator Result<User>(TValue)

// Explicit success (only when implicit is ambiguous)
return Result<User>.Success(user);

// Error returns (all implicit)
return Error.NotFound("User.NotFound", $"User {id} not found");
return Error.Validation("User.Email", "Email is required");
return Error.Conflict("User.Duplicate", "Email already registered");
```

`Result<TValue>` has both implicit operators defined
(`src/Modulus.Mediator.Abstractions/Results/ResultT.cs:47-50`):

```csharp
public static implicit operator Result<TValue>(TValue value) => Success(value);
public static implicit operator Result<TValue>(Error error)  => Failure(error);
```

### Consuming Results with Match

Always use `Match` at the API boundary — never read `.Value` directly from a `Result` that could be failed:

```csharp
var result = await mediator.Send(query, ct);

return result.Match(
    user => Results.Ok(user),
    failure => failure.Errors[0].Type switch
    {
        ErrorType.NotFound   => Results.NotFound(),
        ErrorType.Validation => Results.ValidationProblem(
            failure.Errors.ToDictionary(e => e.Code, e => new[] { e.Description })),
        _                    => Results.Problem()
    });
```

## Records and Interface Markers

All commands, queries, events, and value objects are `record` types. This enforces immutability and structural equality.

```csharp
// Commands — write operations
public record CreateUserCommand(string Email, string Name) : ICommand<User>;
public record DeleteUserCommand(Guid UserId) : ICommand;

// Queries — read operations
public record GetUserQuery(Guid UserId) : IQuery<User>;
public record ListUsersQuery(int Page, int PageSize) : IQuery<IReadOnlyList<User>>;

// Domain events — in-process only
public record UserCreatedDomainEvent(Guid UserId) : IDomainEvent;

// Integration events — cross-module via MassTransit (see the masstransit skill)
public record UserCreatedEvent(Guid UserId, string Email) : IIntegrationEvent;
```

**Why records?** Positional properties, value equality, non-destructive mutation via `with`, and concise syntax.
Mutating a command after creation is a bug — immutability prevents it structurally.

## Sealed Classes and Primary Constructors

Every DI-consuming class must be `sealed`. This prevents accidental inheritance that breaks the DI lifecycle.
Use primary constructors — field declarations are generated by the compiler with no boilerplate.

```csharp
// GOOD — sealed + primary constructor
public sealed class CreateUserHandler(
    IUserRepository repo,
    IMessageBus bus,
    ILogger<CreateUserHandler> logger)
    : ICommandHandler<CreateUserCommand, User>
{
    public async Task<Result<User>> Handle(CreateUserCommand cmd, CancellationToken ct)
    {
        var exists = await repo.ExistsAsync(cmd.Email, ct);
        if (exists)
            return Error.Conflict("User.Duplicate", "Email already registered");

        var user = new User(cmd.Email, cmd.Name);
        await repo.AddAsync(user, ct);
        await bus.PublishAsync(new UserCreatedEvent(user.Id, user.Email), ct);

        return user;
    }
}
```

```csharp
// BAD — unnecessary field declarations and non-sealed
public class CreateUserHandler : ICommandHandler<CreateUserCommand, User>
{
    private readonly IUserRepository _repo;
    private readonly IMessageBus _bus;

    public CreateUserHandler(IUserRepository repo, IMessageBus bus)
    {
        _repo = repo;
        _bus = bus;
    }
    // ...
}
```

Primary constructors expose parameters as synthesized fields — they are `readonly` and cannot be reassigned,
matching the intent of DI-injected dependencies.

## Implicit Conversions

Lean on implicit operators; they make handler bodies readable without losing type safety.

```csharp
// All of these compile to the same IL:
return user;                                 // implicit TValue -> Result<TValue>
return Result<User>.Success(user);           // explicit (only when needed)

return Error.NotFound("X", "msg");           // implicit Error -> Result<User>
return Result<User>.Failure(Error.NotFound("X", "msg"));  // verbose — avoid
```

**WARNING:** Do NOT use implicit conversions inside async lambdas or `Task.FromResult` without care —
the compiler may infer `Task<User>` instead of `Task<Result<User>>`:

```csharp
// BAD — compiler infers wrong type
return Task.FromResult(user);                // Task<User>, not Task<Result<User>>

// GOOD — explicit
return Task.FromResult<Result<User>>(user);

// BEST — await if you have any async call in the method
var user = await repo.FindAsync(id, ct);
return user ?? Error.NotFound("User.NotFound", "...");
```

## Anti-Patterns

### WARNING: Throwing Exceptions for Expected Errors

**The Problem:**

```csharp
// BAD — exception for an expected business condition
public async Task<User> Handle(GetUserQuery query, CancellationToken ct)
{
    var user = await repo.FindAsync(query.UserId, ct);
    if (user is null)
        throw new NotFoundException($"User {query.UserId} not found");
    return user;
}
```

**Why This Breaks:**
1. Exceptions bypass the mediator pipeline — `LoggingBehavior` won't log the failure correctly
2. `UnhandledExceptionBehavior` catches all exceptions and wraps them as `ErrorType.Failure` (500), losing the 404 semantic
3. Every caller must wrap in try/catch, making composition impossible

**The Fix:**

```csharp
// GOOD
public async Task<Result<User>> Handle(GetUserQuery query, CancellationToken ct)
{
    var user = await repo.FindAsync(query.UserId, ct);
    return user ?? Error.NotFound("User.NotFound", $"User {query.UserId} not found");
}
```

### WARNING: Public Setters on Value-Carrying Types

**The Problem:**

```csharp
// BAD — mutable command breaks immutability guarantee
public class CreateUserCommand : ICommand<User>
{
    public string Email { get; set; }
    public string Name { get; set; }
}
```

**Why This Breaks:**
1. Any middleware in the pipeline can mutate the request — creates hidden bugs
2. Breaks structural equality — two "equal" commands may diverge after mutation
3. MOD005 analyzer fires a warning; records fix both issues

**The Fix:**

```csharp
// GOOD — immutable record
public record CreateUserCommand(string Email, string Name) : ICommand<User>;
```

---

### WARNING: Manual Handler Registration

**The Problem:**

```csharp
// BAD — manually wiring handlers in DI
services.AddScoped<ICommandHandler<CreateUserCommand, User>, CreateUserHandler>();
services.AddScoped<IValidator<CreateUserCommand>, CreateUserValidator>();
```

**Why This Breaks:**
1. You will forget to add new handlers — silent runtime failures
2. Source generator already does this at compile time; manual registration creates duplicates
3. Double-registration causes DI ambiguity bugs for `IEnumerable<IValidator<T>>`

**The Fix:**

```csharp
// GOOD — one call, all handlers discovered at compile time
services.AddModulusHandlers();
```
